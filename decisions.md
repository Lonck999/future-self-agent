# Future Self Agent — 待決策清單

> 用途：列出專案落地前需要決定的所有關鍵問題。請在每一項後面填入你的決定，未決定的先留白。

## 1. 人格/目標定義
- [x] 輸入方式：**混合式**——使用者先自由描述，系統再追問補齊時間範圍、衡量標準等關鍵欄位，最後產出結構化摘要供確認。
- [x] 角色數量：**先做單一身分**（MVP），跑通整個迴圈後再擴充支援多重角色。
- [x] 目標粒度：**先只談職業/技能**，之後再擴充納入生活習慣/價值觀層面。

## 2. 調查與規劃邏輯
- [x] 資料來源：**LLM 知識為主**，遇到具體職位/證照等會隨時間變動的資訊時，**才觸發 WebSearch** 補充驗證。
- [x] 現況評估：**讀取使用者現有資料**（履歷/筆記/行事曆歷史），需另外整合資料存取（待第6/9大題確認存取範圍與隱私邊界）。
- [x] 技能差距排序：**必要程度 x 現況落差**，兩者皆高者優先學習，不採複雜加權模型。
- [x] 時間粒度：**固定週期（例：3 個月一期）**，每月底檢視進度、調整下一期，與每月確認機制對齊。

## 3. 狀態與記憶模型
- [x] 存放位置：**專案內本地檔案**（profile / plan / progress-log），Agent 直接讀寫，git 追蹤歷史。
- [x] 單一真相來源格式：**純 JSON**（注意：使用者手動閱讀/編輯體驗較差，之後可考慮加一份 Markdown 唯讀視圖供人看）。
- [x] 紀錄累積：**每日 append 進 progress-log，每月底摘要成月報寫入 plan 歷史區，當月細項日誌歸檔**，避免主檔案無限增長。
- [x] 手動修改對齊：**不做偵測機制**，每次排程執行一律重新讀取當下檔案內容為準。

## 4. 每日規劃流程
- [x] 觸發方式：**完全自動排程**（`/schedule` 設定每日 cron routine，固定時間自動執行）。
- [x] 待辦粒度：**具體可執行任務**（例如「閱讀 XX 章節，1 小時」），確保行事曆事件有實際指引價值。
- [x] 未完成處理：**自動順延到下一天並提高優先度**，連續未完成則留待每月檢查機制提醒，不做每日重新規劃。

## 5. 每月進度確認流程
- [x] 確認依據：**依使用者自己寫的筆記**判斷進度（而非主動問卷、也非系統檢查行事曆完成率，因 Calendar 無完成勾選功能）——呼應第2大題「讀取現有資料」的決定，月度檢查時讀取使用者筆記內容來評估。
- [x] 未達標處理：**只提出調整建議，使用者確認後才套用**，系統不自行修改計畫。
- [x] 月度報告：**主動推送通知**提醒使用者查看進度報告。

## 6. Google Calendar 整合
- [x] 授權：已透過 Claude.ai 整合的 `Google_Calendar` MCP 工具具備授權，不需額外設定 OAuth。
- [x] 事件結構：**每項任務建立獨立事件**，方便個別檢視與調整。
- [x] 衝突避免：**用唯一 ID 比對更新**——plan/progress-log 記錄任務對應的 event ID，調整時用 `update_event`，不整批刪除重建。
- [x] 行事曆區隔：**建立獨立的「未來自己」行事曆**。**技術限制**：現有 `Google_Calendar` MCP 工具沒有「建立新日曆」功能，只能在既有日曆內操作事件。決定：**使用者手動在 Google Calendar 建立新日曆**，把日曆名稱/ID 提供給系統寫入 `plan.json.calendar`，系統用 `list_calendars` 確認後續用該 ID 操作事件。

## 7. 自動化執行機制
- [x] 觸發機制：**Claude Code 排程 cron routine**（`/schedule`），每日、每月各設一個 routine，與第4大題一致。
- [x] 失敗處理：**自動重試幾次**（注意：因第6大題已用唯一 ID 比對更新事件，重試時不會重複建立行事曆事件，可降低重試的副作用風險；仍需記錄失敗日誌，多次重試仍失敗則通知使用者）。
- [x] 中斷/暫停：**要支援**，`plan.json` 加 `status: active/paused` 欄位，排程執行前先檢查狀態，paused 時跳過該次執行，cron routine 本身不取消，方便之後恢復。

## 8. 互動與通知體驗
- [x] 查看介面：**Google Calendar（每日任務）+ 月度推送通知（摘要報告）**，不額外打造儀表板網站。
- [x] 調整原則：**依情境分級處理**——
  - 日常/routine 層級（如未完成任務順延、優先度微調，見第4大題）：**自動執行**，不需每次徵求同意。
  - 結構性/重大層級（如改變計畫期長、調整目標技能方向、月度未達標的計畫調整，見第5大題）：**先提建議，使用者明確同意才套用**。

## 9. 多人/隱私考量
- [x] 使用者範圍：**先做單人系統**，只服務使用者自己一個帳號。
- [x] 隱私邊界：**所有個人資料（履歷/筆記/行事曆歷史）只留在專案本地檔案或使用者自己的 Google 帳號內**，不外傳第三方；WebSearch 僅查公開資訊，不帶出個人資料。
